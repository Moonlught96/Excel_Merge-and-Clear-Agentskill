# 用户名与昵称哈希兜底设计

## 1. 目标

在现有 `哈希ID` 去标识化能力上增加确定性的显示名兜底：当已确认平台的原始数据不存在可用的稳定账号 ID 时，工具可以按平台固定配置选择用户名或昵称，生成稳定的 `哈希ID`。

该能力用于提高缺少公开账号 ID 的评论数据的可关联率。转换过程只由固定配置、字符串规范化和 HMAC-SHA256 工具完成，不使用 AI 识别、语义判断或相似度匹配。

## 2. 与既有设计的关系

本设计补充并修订 `2026-07-16-hash-id-pseudonymization-design.md` 中“只处理真实用户 ID、不得回退昵称”的边界。其余项目隔离、平台隔离、DPAPI 密钥保存、输出列、安全日志和清洗流程约束保持不变。

兼容性要求：

- 稳定账号 ID 的字段选择、规范化和 HMAC 消息格式保持不变。
- 已经基于稳定账号 ID 生成的哈希结果不得因本功能变化。
- 显示名使用新的身份类型命名空间，不能与稳定账号 ID 的哈希空间发生碰撞。

## 3. 适用边界

### 3.1 优先级

每个平台按以下固定顺序选择身份来源：

1. 查找该平台配置中的 `user_id_headers`，按配置顺序选择第一个实际存在的稳定账号 ID 列。
2. 只有不存在任何已登记稳定账号 ID 列时，才查找 `display_name_headers`。
3. 按 `display_name_headers` 的配置顺序选择第一个实际存在的用户名或昵称列。
4. 两类字段都不存在时，`哈希ID` 保持空白。

只要源表存在已登记稳定账号 ID 列，就不能因部分行 ID 为空而逐行回退到昵称。整份工作表必须使用同一种身份来源，避免一个批次混用两种身份强度。

### 3.2 去标识化强度

- 稳定账号 ID 哈希属于较强的项目内关联标识。
- 用户名或昵称哈希属于弱关联标识，只能用于初步分析。
- 昵称修改会使同一用户产生新哈希。
- 不同用户使用相同用户名或昵称时可能被合并为同一哈希。
- 本能力是去标识化措施，不宣称实现法律意义上的匿名化，也不替代合法来源、目的限定、最小必要、访问控制和留存期限管理。

## 4. 平台固定映射

配置文件为每个平台增加有序的 `display_name_headers`：

| 平台命名空间 | 稳定账号 ID 优先字段 | 显示名兜底字段（按优先级） |
|---|---|---|
| `youtube` | `author_channel_id`, `authorChannelId`, `Author Channel ID` | `author` |
| `xiaohongshu` | `用户ID` | `用户名称` |
| `bilibili` | 暂无 | `username` |
| `tiktok` | 暂无 | `用户名`, `昵称` |
| `taobao` | 暂无 | `用户名称`, `用户名` |
| `jd` | 暂无 | `用户名` |

约束：

- TikTok 的 `用户身份` 含义不明确，不得作为显示名来源。
- 评论 ID、父评论 ID、视频 ID、帖子 ID、主页 URL、IP 和设备信息不得作为显示名兜底。
- 新增平台或表头别名必须由用户确认后写入配置，不得通过 AI 猜测。
- 同一表中出现多个显示名候选列时严格按配置顺序选择，不做内容质量比较。

## 5. 显示名规范化

显示名按以下确定性规则规范化：

1. CSV 继续按文本读取。
2. 删除字符串首尾空白。
3. 保留大小写、内部空白、标点、Unicode 字符和前导零。
4. 不做大小写折叠、全半角转换、繁简转换、音译、语言识别或模糊匹配。
5. 空值或删除首尾空白后为空的值输出空哈希。
6. 非字符串值按现有 ID 规范化的安全规则处理；布尔值、错误值和公式形式视为无效并明确报错，错误信息不得包含原值。

由此保证：同一项目、同一平台内，规范化后文本完全相同的用户名或昵称必须得到完全相同的 `哈希ID`。该结果不受源字段名是“用户名”还是“昵称”影响。

不同平台继续使用独立平台命名空间。跨平台恰好同名的显示名必须得到不同哈希，避免错误推断为同一自然人。

## 6. 哈希消息与兼容性

### 6.1 稳定账号 ID

稳定账号 ID 继续使用既有 `bazhuayu-hash-id-v1` 消息格式，不增加任何字段，不改变历史结果：

```text
message = encode(algorithm_version, project_id, key_version, platform_namespace, normalized_user_id)
hash_id = hex(HMAC-SHA256(project_key, message))
```

### 6.2 用户名或昵称

显示名使用明确的 `display_name` 身份类型段：

```text
message = encode(
  algorithm_version,
  project_id,
  key_version,
  platform_namespace,
  "display_name",
  normalized_display_name
)
hash_id = hex(HMAC-SHA256(project_key, message))
```

身份类型只区分 `account_id` 既有路径和 `display_name` 新路径。显示名消息不包含具体源表头名称，因此 TikTok 的“用户名”和“昵称”只要规范化文本相同，就会产生相同哈希。

输出仍为完整 64 位小写十六进制字符串，不提供解密、反查或原始值映射表。

## 7. 配置与代码模型

平台配置模型增加：

```json
{
  "namespace": "bilibili",
  "aliases": ["B站", "哔哩哔哩", "bilibili"],
  "user_id_headers": [],
  "display_name_headers": ["username"]
}
```

身份字段选择结果必须包含：

- `source_header`
- `source_column`
- `identity_type`：`account_id` 或 `display_name`

代码应使用单一的配置驱动选择器完成优先级判断，不在标准化脚本中硬编码平台分支。项目工具与 Skill 内打包脚本、配置必须同步。

## 8. 标准化流程行为

1. Agent 继续使用用户已确认的研究项目和数据来源调用工具。
2. 标准化工具加载对应项目密钥和平台映射。
3. 先选择稳定账号 ID；不存在时选择显示名兜底。
4. 每个非空值只在内存中规范化并生成 `哈希ID`。
5. 原始账号 ID、用户名和昵称均不得进入标准化输出。
6. 标准化输出继续只保留固定九列，`哈希ID` 位于 `产品名` 后。
7. 标准化完成后仍需返回用户确认，再进入 KOL 清理词确认与清洗。

用户名或昵称只参与哈希转换，不改变评论内容、日期、点赞数、子评论或其它业务字段。

## 9. 摘要与隐私

标准化摘要允许增加：

- `identity_type`
- 选中的源表头名称
- 成功生成哈希的数量
- 空身份值数量
- 平台、项目 ID、算法版本和密钥指纹

摘要、日志和错误信息不得记录：

- 原始账号 ID
- 原始用户名或昵称
- 逐行哈希明细
- 项目密钥或加密密钥内容

当使用显示名兜底时，摘要必须明确标记为 `display_name`，以便后续分析区分弱关联数据，不能把它描述为稳定账号 ID。

## 10. 错误处理

- 已登记身份列存在但项目配置或平台参数缺失时必须停止，不能静默输出空哈希。
- 同一优先级配置出现重复表头、配置类型错误或空表头时必须在加载配置时失败。
- 单元格转换失败时只报告工作表、行号、字段名、身份类型和失败类型，不报告原始值。
- 没有任何已登记身份列不是错误，`哈希ID` 保持空白并在摘要记录原因。

## 11. 测试要求

### 11.1 单元测试

- 同一项目、同一平台、相同规范化显示名生成相同哈希。
- 同一文本分别来自“用户名”和“昵称”时生成相同哈希。
- 相同显示名在不同项目或不同平台生成不同哈希。
- 显示名文本与稳定账号 ID 文本相同时生成不同哈希。
- 显示名保留大小写和内部空白；只删除首尾空白。
- 稳定账号 ID 存在时优先使用，不选择显示名。
- 稳定账号 ID 列存在但某行为空时，该行保持空白，不逐行回退显示名。
- 配置顺序决定多个显示名候选列的选择结果。
- TikTok `用户身份`、评论 ID、父评论 ID 和 URL 不得被选择。
- 摘要和错误信息不包含原始显示名。
- 既有稳定账号 ID 哈希固定向量保持不变。

### 11.2 集成测试

- B站 `username` 能在标准化输出中生成非空 `哈希ID`。
- TikTok 优先使用 `用户名`，缺列时再使用 `昵称`。
- YouTube 有 `author_channel_id` 时不使用 `author`；缺少账号 ID 列时使用 `author`。
- 小红书有 `用户ID` 时不使用 `用户名称`；缺少用户 ID 列时使用 `用户名称`。
- 京东、淘宝按固定配置生成显示名哈希。
- 同一项目不同批次中相同平台、相同显示名稳定一致。
- 标准化后的清洗过程不改写或丢失 `哈希ID`。
- 原始输入文件校验值不变。

### 11.3 Skill 与回归验证

- 同步更新项目配置、工具、测试、README、AGENTS 和 Skill references。
- 同步打包后的 `scripts/` 与 `config/`，执行同步一致性检查。
- 执行完整项目测试和 Skill 独立复制运行测试。
- 使用固定测试密钥验证，不读取或泄露真实项目密钥。

## 12. 当前 ScreenBar 批次迁移

实现并通过测试后：

1. 为用户已确认的 `ScreenBar十周年专案` 初始化或加载 DPAPI 保护的项目密钥。
2. 从本轮 B站“回复前缀已清理”的合并表重新执行标准化，不修改原始输入与合并文件。
3. 使用 `username` 生成显示名类型的 `哈希ID`。
4. 覆盖本轮尚未确认的标准化输出前，先验证行数、九列顺序、非空哈希数量和相同用户名的一致性。
5. 返回新的标准化总表供用户确认，确认后再继续既有清洗流程。

## 13. 验收标准

1. 稳定账号 ID 优先，只有整表缺少已登记账号 ID 列时才启用显示名兜底。
2. 同一项目、同一平台、规范化后相同的用户名或昵称始终得到相同哈希。
3. 用户名列与昵称列中的相同文本得到相同哈希。
4. 不同平台、不同项目或不同身份类型无法通过相同哈希直接关联。
5. 既有稳定账号 ID 哈希结果保持不变。
6. 原始账号 ID、用户名、昵称和项目密钥不进入标准化输出、清洗输出、日志、摘要或仓库。
7. 全流程由确定性工具执行，不接入 AI 判断。
8. 合并、标准化、清洗、文件命名和中间产物清理的既有回归测试全部通过。
