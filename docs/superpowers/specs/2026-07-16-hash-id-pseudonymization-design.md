# 哈希ID去标识化设计

## 1. 目标

在现有 Excel/CSV 评论标准化流程中增加 `哈希ID` 标准列。工具读取平台提供的真实用户 ID，在内存中使用项目级密钥执行确定性 HMAC-SHA256 转换，只把不可逆的哈希结果写入标准化输出。

本功能用于在同一研究项目、同一平台内关联同一用户的多次评论，同时降低原始用户 ID 在标准化文件、清洗文件和后续分析文件中泄露的风险。

## 2. 明确边界

### 2.1 去标识化而非匿名化

- `哈希ID` 可以稳定关联同一用户，因此按去标识化数据管理，不宣称为不可关联的匿名数据。
- 哈希处理不替代合法来源、目的限定、最小必要、访问控制、合理留存期限等管理要求。
- 原始输入文件保持不变，仍可能包含原始用户 ID；本功能只保证原始 ID 不进入标准化输出及其后续产物。

### 2.2 只处理真实用户 ID

- `哈希ID` 只接受已登记平台中的真实用户或账号 ID 字段。
- 用户名、昵称、账号显示名、主页地址、IP、设备 ID 不得作为用户 ID 的替代值。
- 评论 ID、父评论 ID、视频 ID、帖子 ID 不得映射为 `哈希ID`。
- 源文件没有可用用户 ID 时，`哈希ID` 保持空白，不推断、不拼接、不回退到昵称。

### 2.3 完全确定性

- 表头识别只使用版本化配置中的平台专属别名映射。
- 平台识别只使用用户已确认的数据来源和固定平台名称映射。
- 项目边界只使用用户首次确认后创建的项目配置。
- ID 转换只使用固定规范化规则和标准密码学函数，不调用语言模型或语义分类。

## 3. 标准数据契约

标准化输出列调整为：

1. `评论日期`
2. `评论内容`
3. `产品名`
4. `哈希ID`
5. `点赞数`
6. `子评论数/追评数`
7. `一级评论`
8. `二级评论`
9. `三级评论`

`哈希ID` 是必保留但允许为空的标准列。标准化和清洗后的 `.xlsx`、`.csv` 均使用相同列顺序。

原始用户 ID 不得出现在标准化输出、清洗输出、删除日志、摘要、错误信息或调试信息中。

## 4. 项目身份与密钥

### 4.1 项目确认

- 用户首次处理新研究项目时明确提供或确认项目名称。
- 项目名称确认与现有“产品名、数据来源、输出命名”确认合并，一轮流程只确认一次。
- 文件名和父文件夹名可以提供候选名称，但不能自动成为最终项目身份。
- 用户已经声明后续出现新项目时会明确告知；未声明新项目时，工具只可复用当前明确选中的项目配置。
- 找不到项目配置、项目名称不唯一或出现多个候选配置时必须停止，不得猜测。

### 4.2 项目配置

每个项目创建不含秘密值的元数据：

```json
{
  "schema_version": 1,
  "project_id": "随机UUID",
  "project_name": "用户确认的项目名称",
  "key_version": 1,
  "key_fingerprint": "密钥指纹",
  "created_at": "北京时间ISO时间"
}
```

`project_id` 只标识研究项目，不包含产品名、平台用户信息或原始文件路径。

### 4.3 密钥生成和保存

- 每个项目使用密码学安全随机源生成独立的 256 位密钥。
- Windows 默认使用 DPAPI 将密钥加密后保存在当前用户的本地应用数据目录。
- 项目元数据和加密密钥不得保存在 Skill 目录、仓库目录、输出目录或原始数据目录。
- 项目密钥、明文密钥、环境变量值不得写入 Git、表格、日志、摘要或对话回复。
- 自动化测试只使用测试进程注入的固定测试密钥，不读取真实项目密钥。
- Skill 在非 Windows 环境运行时必须支持显式安全密钥提供器；没有可用密钥提供器时停止，不得把明文密钥降级写入项目目录。

### 4.4 密钥生命周期

- 同一项目后续批次继续使用相同项目配置和相同密钥版本。
- 新项目必须生成新的项目 ID 和密钥。
- 密钥轮换会产生新的哈希结果；轮换前必须明确提示旧批次和新批次无法直接关联。
- 密钥丢失后不能恢复既有哈希对应关系，不得根据旧哈希反推或重建原始 ID。

## 5. 平台隔离

平台名称通过固定配置规范为稳定命名空间，例如：

| 用户确认的数据来源 | 规范平台命名空间 |
|---|---|
| B站、哔哩哔哩、bilibili | `bilibili` |
| YouTube、YouTube Shorts | `youtube` |
| TikTok、TTCommentExporter | `tiktok` |
| 小红书 | `xiaohongshu` |
| 淘宝 | `taobao` |
| 京东 | `jd` |

- 未登记的平台名称必须停止并让用户确认新增映射。
- 同一原始用户 ID 在不同平台必须生成不同哈希。
- 产品名、文件名、评论内容和抓取时间不参与平台判断。

## 6. 用户 ID 字段映射

新增平台专属 `identity_sources` 配置，结构包含：

- 规范平台命名空间。
- 已确认的真实用户 ID 表头别名。
- 明确禁止使用的评论 ID、父评论 ID、用户名和昵称表头。
- 同一平台出现多个候选用户 ID 列时的固定优先级。

映射规则：

1. 只在当前已确认平台的映射范围内查找用户 ID。
2. 命中一个已登记字段时读取该列。
3. 命中多个字段时按配置中的固定优先级选择，并在摘要中只记录选中的表头名称。
4. 没有命中时输出空的 `哈希ID` 列。
5. 命中未登记或含义冲突的 ID 表头时停止并要求确认映射。
6. 全局名称如 `id`、`cid` 不得脱离平台上下文直接认定为用户 ID。

首轮实现前必须使用现有各平台样本核对真实字段含义。只有经用户确认的别名才进入正式配置。

## 7. ID规范化与HMAC算法

### 7.1 原始 ID 规范化

- CSV 按现有兼容层以文本读取。
- 字符串只删除首尾空白，保留大小写、内部空白和前导零。
- Excel 整数单元格转换为无小数点的十进制文本。
- Excel 中值为整数的浮点单元格按整数文本处理，例如 `123.0` 规范为 `123`。
- 非整数数值、布尔值、错误值或公式形式的用户 ID 视为无效输入，停止并报告工作表、行号和字段名，但不报告原始值。
- 空值和全空白值输出空哈希。

### 7.2 消息格式

HMAC 输入使用带版本号和长度边界的确定性编码，避免简单字符串拼接产生歧义：

```text
version = "bazhuayu-hash-id-v1"
message = encode(version, project_id, platform_namespace, normalized_user_id)
hash_id = hex(HMAC-SHA256(project_key, message))
```

编码函数对每一段使用 UTF-8 字节长度前缀。输出为完整 64 位小写十六进制字符串，不截断。

### 7.3 稳定性保证

- 相同项目、相同密钥版本、相同平台、相同规范化用户 ID：结果相同。
- 项目、密钥版本、平台或用户 ID 任一不同：结果应不同。
- 算法不提供解密接口、反查接口或原始 ID 映射表。

## 8. 标准化流程集成

1. 用户提供一个或多个原始 Excel/CSV。
2. 现有流程确认项目名称、产品名、数据来源和输出命名。
3. 工具加载或初始化用户确认的项目配置。
4. 多文件流程先按既有规则合并，原始文件保持不变。
5. 标准化工具根据已确认平台查找用户 ID 字段。
6. 每个非空用户 ID 在内存中规范化并转换为 `哈希ID`。
7. 标准化输出只保留九个标准列，不写出原始 ID。
8. 用户确认标准化结果后，继续现有 KOL 清理词确认与清洗流程。
9. 清洗完成后仍按既有规则只保留清洗后的 `.xlsx` 和 `.csv`。

本功能不得改变既有评论内容清洗、子评论清洗、表头排序、文件命名和中间产物清理规则。

## 9. 命令与Agent调用边界

标准化工具增加显式参数：

- `--project-name` 或 `--project-id`：选择已确认项目。
- `--platform`：传入已确认的数据来源并解析为规范平台命名空间。
- `--initialize-project`：只在用户首次确认新项目时创建配置和密钥。

Agent 负责把用户已确认的信息传给工具，不得自行生成项目名称或平台映射。用户不需要手动打开命令行。

若源文件不存在用户 ID 列，工具仍生成空 `哈希ID` 列；此时不因缺少 ID 而阻止标准化。若检测到已登记用户 ID 列但没有项目配置或平台参数，则必须失败，不得静默删除该 ID。

## 10. 摘要与错误处理

标准化摘要允许记录：

- 项目 ID、项目名称和密钥指纹。
- 规范平台命名空间。
- 选中的源用户 ID 表头名称。
- 成功生成哈希的数量和空值数量。
- 算法及配置版本。

摘要和错误信息不得记录：

- 原始用户 ID。
- 哈希ID逐行明细。
- 项目密钥或加密密钥内容。
- 昵称、用户名、IP 或主页地址。

任何单元格转换失败都必须明确报告工作表、行号、字段名和失败类型，不能静默写空。

## 11. 验证要求

### 11.1 单元测试

- 相同项目、平台和 ID 生成相同哈希。
- 不同项目、平台或 ID 生成不同哈希。
- 字符串 ID 保留前导零和大小写。
- 整数与等值整数浮点数产生相同规范值。
- 空 ID 输出空值。
- 用户名、昵称、评论 ID 和父评论 ID 不得生成哈希。
- 未登记平台、含义冲突表头、无密钥和无效 ID 必须失败。
- 输出、摘要和错误信息不包含原始 ID 或密钥。

### 11.2 集成测试

- `.xlsx`、`.xlsm`、`.csv` 输入均可生成九列标准输出。
- 同一项目多批文件中的相同平台用户得到相同哈希。
- 同一原始用户 ID 在 B站和 YouTube 中得到不同哈希。
- 标准化后继续运行现有清洗工具，`哈希ID` 不被改写或丢失。
- 清洗后的 `.xlsx` 和 `.csv` 列顺序一致。
- 原始输入文件内容和校验值不变。

### 11.3 Skill同步验证

- 同步更新项目工具、配置、测试、README 和 AGENTS 规则。
- 同步更新 `skills/product-user-comment-data-merge-cleaning/` 中的脚本、配置、references、assets 和 `SKILL.md`。
- 执行项目测试、Skill 包完整性检查和复制到独立目录后的运行测试。
- 测试使用临时项目和临时密钥，测试结束后删除临时密钥材料。

## 12. 验收标准

只有同时满足以下条件才算完成：

1. 标准化输出固定包含 `哈希ID` 且列顺序正确。
2. 相同项目、同平台、同用户 ID 的结果稳定一致。
3. 跨项目和跨平台无法通过相同哈希直接关联。
4. 原始用户 ID 和项目密钥未出现在任何输出、日志、摘要或仓库文件中。
5. 没有真实用户 ID 的数据保持空列，工具不进行身份猜测。
6. 既有合并、标准化、清洗和中间文件清理流程继续通过回归测试。
7. 完整 Skill 文件夹可独立复制运行，并保持相同安全边界。
