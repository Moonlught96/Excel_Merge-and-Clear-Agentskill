# 产品用户评论数据合并与清洗 Skill

功能描述：为抓取的大量用户评论数据进行文档合并、标准化和清洗工作，并输出为 XLSX 与 CSV 格式文档。

输入描述：目前输入不受限制，支持 Excel 与 CSV 文件。

- 读取一个或多个 `.xlsx`、`.xlsm` 或 `.csv` 文件。
- 遍历所有工作表。
- 先把表头整理为固定列：评论日期、评论内容、产品名、哈希ID、点赞数、子评论数/追评数、一级评论、二级评论、三级评论。
- B站合并表在标准化前可先按固定规则去掉 `回复@xxx：` 或 `回复 @xxx:` 前缀，只保留冒号后的真实评论内容。
- 删除不进入后续流程的风险列，例如昵称、IP 数据。
- 从第 2 行开始读取评论内容。
- 删除空白、过短、占位文案、包含 KOL 清理词、包含固定清理词的行；固定清理词是在原有“链接”基础上追加，不覆盖原规则。
- 删除同一工作表内重复评论。
- 对 `一级评论`、`二级评论`、`三级评论` 做精确去重，并清空首尾空白后长度小于等于 5 的子评论单元格；不删除整行，也不改动 `评论内容`。
- 原始文件不改动，另存为新的 `.xlsx` 和第一个工作表的 `.csv`。
- 同时输出删除日志，方便核对每一行被删除的原因。
- 清洗后的 `.xlsx` 和 `.csv` 生成并核对成功后立即删除合并总表、回复前缀清理表、标准化总表、清洗日志和摘要等过程产物，默认最终只保留这两个清洗结果文件。

CSV 输入会先通过确定性兼容层读取，单元格值按文本保留，不自动推断数字、日期、ID 或时间戳类型。整个处理过程是纯工具规则处理，不接入 AI 判断评论内容。AI 只负责帮你调用工具、核对输出文件是否生成，不参与“是否删除某条数据”的判断。
Excel 输入中的公式会按公式文本保留；合并、B站回复前缀处理和标准化不会因为缺少缓存计算值而把公式单元格变成空白。

## 快速运行

```powershell
python tools/standardize_excel_headers.py "D:\path\input.xlsx-or.csv" --output-dir "D:\path\output"
python tools/clean_excel_comments.py "D:\path\output\标准化后的文件.xlsx" --target-header "评论内容" --clean-word "KOL清理词1" --clean-word "KOL清理词2"
```

如果本机没有全局 `python`，让 Codex 运行即可。Codex 会使用内置 Python 环境。

B站评论数据如果已经合并成原始总表，标准化前先运行这个确定性前缀清理步骤：

```powershell
python tools/strip_bilibili_reply_prefixes.py "D:\path\合并总表.xlsx" --output "D:\path\合并总表_回复前缀已清理.xlsx"
python tools/standardize_excel_headers.py "D:\path\合并总表_回复前缀已清理.xlsx" --output "D:\path\标准化总表.xlsx"
```

这个步骤只处理 `content` 或 `评论内容` 列中开头匹配 `回复@xxx：`、`回复 @xxx：`、`回复@xxx:`、`回复 @xxx:` 的固定文本前缀；不移动回复行，不推断父子层级，不新增列，也不删除行。

清理词是可选项；不填写时，不执行 KOL 清理词包含判断：

```powershell
python tools/standardize_excel_headers.py "D:\path\input.xlsx-or.csv" --output-dir "D:\path\output"
python tools/clean_excel_comments.py "D:\path\output\标准化后的文件.xlsx" --target-header "评论内容"
```

## 配置

默认规则在：

```text
config/comment-cleaner.json
```

常用可改项：

- `target_column`: 评论所在列，默认 `3`。
- `target_header`: 评论所在表头。标准化后使用 `评论内容`。
- `first_data_row`: 数据起始行，默认 `2`。
- `min_trimmed_length`: 清理首尾空白后的最短保留长度，默认小于 `8` 删除，也就是长度小于等于 `7` 删除。
- `delete_exact_texts`: 完全等于这些文案时删除。
- `delete_contains_texts`: 包含这些固定文案时删除。当前清单保留原有“链接”，并追加“凑字数”“为了金币”“暂无评价”“蹲一个”“交朋友”等固定词。
- `delete_contains_case_insensitive_texts`: 大小写不敏感的固定词；当前用于“加v”，所以“加v”和“加V”都会删除。
- `delete_random_alnum_without_chinese`: 完全不含中文字符且命中随机英文/数字堆砌阈值时删除；该判断只用确定性正则和阈值，不接入 AI。
- `subcomment_deduplicate_headers`: 需要做子评论精确去重的列，默认是 `一级评论`、`二级评论`、`三级评论`。
- `subcomment_min_trimmed_length`: 子评论清理首尾空白后的最短保留长度，默认小于 `6` 清空，也就是长度小于等于 `5` 清空。
- `duplicate_keep`: 重复评论保留哪一条。默认 `last`，匹配截图里从最后一行往前遍历的行为。

子评论去重只做完全一致文本匹配，不做 AI 判断、语义相似判断或模糊匹配。默认保留最后一次出现的子评论，前面重复出现的对应单元格会被清空；子评论长度清理只清空对应单元格；主评论列 `评论内容` 和整行其它数据不移动、不删除。

`KOL清理词1`、`KOL清理词2` 不写死在配置里，由你每次运行时输入。传入几个清理词，就按几个清理词判断；不传入则跳过这类判断。

表头标准化规则在：

```text
config/header-standardizer.json
```

默认只保留并排序为：

1. `评论日期`
2. `评论内容`
3. `产品名`
4. `点赞数`
5. `子评论数/追评数`
6. `一级评论`
7. `二级评论`
8. `三级评论`

`评论时间` 会规范成 `评论日期`；`timestamp` 会按北京时间（UTC+8）转换成 `YYYY-MM-DD` 日期，只保留年月日，不保留时分秒；`评论内容` 或 `content` 会规范成 `评论内容`；淘宝表头 `评论日期与产品` 会按固定格式拆分：开头日期进入 `评论日期`，`已购：` 后面的内容进入 `产品名`；京东等独立表头 `购买产品` 会直接进入 `产品名`。`点赞量` 或 `like_count` 会规范成 `点赞数`；`子评论数`、`子评论数（追评数）`、`追评数` 或 `评论数` 会规范成 `子评论数/追评数`；`一级评论内容` 或 `追评` 会规范成 `一级评论`；`引用的评论内容` 或 `二级评论内容` 会规范成 `二级评论`。昵称、IP 数据、`rpid`、`parent_rpid`、`username`、`ip_location` 以及其它不在保留清单内的列不会进入标准化后的文件；其中 `parent_rpid` 是父评论 ID，不会映射为 `子评论数/追评数`。若源文件没有 `产品名`、`一级评论`、`二级评论` 或 `三级评论`，对应列会保留为空，方便后续合并表头一致。

淘宝 `评论日期与产品` 的拆分只使用固定规则，不用 AI：识别开头的 `YYYY年M月D日`、`YYYY/M/D` 或 `YYYY-M-D` 作为评论日期；日期后如有 `已购：`，则把后续文本作为产品名。

如果抓取源换了表头名称，但实际字段还是同一类数据，不用 AI 识别。把新的源表头加入 `config/header-standardizer.json` 的 `aliases` 后，工具会按固定映射排序。`子评论数/追评数` 是标准输出必保留列，但源文件没有对应表头时会保留空列。遇到其它没登记过的必需表头时，工具会停止并列出当前文件的实际表头，等待你确认映射关系。

后续补充别名映射时，可以直接按这个格式告诉 Codex：

```text
标准列“评论日期”的别名新增：评价时间
标准列“产品名”的别名新增：购买商品、商品标题
标准列“点赞数”的别名新增：有用数、点赞
```

Codex 只会把这些名称加入对应标准列的 `aliases`，不会改标准列顺序，也不会用 AI 猜未确认的列。

## 输出

过程文件默认保存在原 Excel 同目录：

- `日期_原文件名.standardized.xlsx`
- `日期_原文件名.standardized.summary.json`
- `日期_原文件名.cleaned.xlsx`
- `日期_原文件名.cleaned.csv`
- `日期_原文件名.deletions.csv`
- `日期_原文件名.summary.json`

清洗后的 `.xlsx` 和 `.csv` 生成并核对成功后立即调用 `tools/cleanup_intermediate_outputs.py`，删除本次流程生成的合并总表、回复前缀清理表、标准化总表、清洗日志、清洗摘要及其它过程摘要文件，不再追加清理确认步骤，默认最终只保留：

- `YYYYMMDD_产品名_数据来源_清洗后总表.xlsx`
- `YYYYMMDD_产品名_数据来源_清洗后总表.csv`

不会删除原始输入文件，也不会扫描文件夹批量删除。如果你明确要求核对日志，Codex 才会保留清洗日志和摘要。

## 适合 Codex 的用法

你以后可以直接说：

```text
用八爪鱼 Excel 清洗工具清理这个文件，清理词是 XXX 和 YYY。
```

Codex 就可以按这个工具执行、检查输出，并把结果路径返回给你。

也可以说：

```text
用八爪鱼 Excel 清洗工具清理这个文件，不填 KOL 清理词。
```

如果你提供了一个或多个 KOL 清理词，Codex 会先确认：

```text
是否已经提供完成所有 KOL 清理词？你确认后我再进行清洗。
```

你确认后，才会把这些清理词传入工具执行清洗。

## 多文件合并

如果有多份八爪鱼导出的 Excel/CSV 文件，可以直接把所有需要合并的文件发给 Codex，并说：

```text
合并这些 Excel/CSV 文件。
```

Codex 只会合并你明确提供的文件，不会扫描整个文件夹。CSV 文件会通过兼容层按文本读取。合并过程不接入 AI，不做内容判断，只按工具规则复制表格数据。

合并之前，Codex 会先确认一次产品名和数据来源。确认后，三个主要文件会按同一组产品名和数据来源自动命名，只替换步骤名：

```text
YYYYMMDD_产品名_数据来源_合并总表.xlsx
YYYYMMDD_产品名_数据来源_标准化总表.xlsx
YYYYMMDD_产品名_数据来源_清洗后总表.xlsx
```

例如：

```text
20260707_ScreenBar Halo2_淘宝评论数据_合并总表.xlsx
20260707_ScreenBar Halo2_淘宝评论数据_标准化总表.xlsx
20260707_ScreenBar Halo2_淘宝评论数据_清洗后总表.xlsx
```

产品名和数据来源会先从文件名、父级文件夹名和表格里的固定产品列中按规则查找；找不到或有多个候选时，Codex 会先问你。这个确认在一个流程中只做一次，后续合并、标准化、清洗不会重复确认命名。

如果你给了多个原始 Excel 文件，Codex 会在展示产品名、数据来源和三步输出文件名时一次性问：

```text
请确认以上产品名、数据来源和文件命名是否正确，并确认是否可以进入合并流程。
```

你确认后才会生成合并总表。如果当前只收到 1 个文件，Codex 不会默认跳过合并，而是先问：

```text
当前只收到 1 个文件，请确认是否只有这一个文件需要处理？你确认后我将跳过合并，直接进入标准化。
```

你确认确实只有一个文件后，Codex 才会跳过合并步骤，直接生成标准化总表。

多文件流程会先合并原始输入文件，生成一个新的原始合并总表；Codex 会先把原始合并总表发给你，并确认是否已经提供并合并完所有需要合并的表格。你确认后，Codex 才会对这个合并总表执行表头标准化，生成一个新的标准化合并总表。原始输入文件不会改动，原始合并总表也不会被标准化步骤覆盖。

如果数据来源是 B站，Codex 会在合并完成并得到你确认后、标准化之前，先对原始合并总表生成一个“回复前缀已清理”的新工作簿。这个步骤只去掉固定的 `回复@xxx：` / `回复 @xxx:` 前缀，保留同一行和同一列中的真实评论内容，不做回复归位。

表头标准化不是只改表头文字，而是把匹配到的源表头和其下整列数据一起移动到标准列位置；不在保留清单内的列才不会进入标准化输出。

合并规则：

- 支持 `.xlsx`、`.xlsm` 和 `.csv`。
- 先合并原始输入文件，确认所有文件都已合并后，再对合并总表生成独立的标准化 Excel 文档。
- 按你提供文件的顺序合并。
- 新建一个独立的 Excel 文档/工作簿文件作为合并总表，不是在原 Excel 里新建子表。
- 不会在原表格基础上合并，原始输入表格保持不动。
- 遍历每个文件的所有工作表。
- 每个工作表第 1 行作为表头。
- 总表只保留第一份表头。
- 从第 2 行开始追加数据。
- 完全空白行跳过。
- 如果后续工作表表头和第一份表头不一致，停止合并并报错。

如果你说：

```text
合并这些 Excel 文件后再清洗。
```

Codex 会先合并原始输入文件，生成原始合并总表，然后停下来确认：

```text
是否已经提供并合并完所有需要合并的表格？你确认后我再进行标准化。
```

合并完成后，Codex 会先把新建的原始合并总表发给你。你确认所有文件都已经合并后，Codex 才会执行标准化，并把标准化后的表格发给你确认。你确认标准化后的表格可以进入清洗流程后，Codex 才会按固定清洗流程询问 KOL 清理词；如果你提供了清理词，Codex 还会再次确认清理词是否已经给全，确认后才执行清洗。

清洗后的 `.xlsx` 和 `.csv` 经你确认成功后，Codex 会清理上述中间文件和默认日志以节省空间，只保留最终清洗结果。

## 哈希ID伪名化

标准化阶段固定生成 `哈希ID` 列。整张工作表优先选择已登记的稳定账号 ID 列；只有不存在已登记账号 ID 列时，才按 `config/hash-id.json` 的平台顺序选择显示名兜底。选择以整张工作表为单位，不按行切换；账号 ID 列存在但某行为空时，该行 `哈希ID` 保持为空。

当前精确映射为：

- YouTube：账号 ID 依次为 `author_channel_id`、`authorChannelId`、`Author Channel ID`；兜底为 `author`。
- 小红书：账号 ID 为 `用户ID`；兜底为 `用户名称`。
- B站：兜底为 `username`。
- TikTok：依次兜底为 `用户名`、`昵称`，绝不使用 `用户身份`。
- 淘宝：依次兜底为 `用户名称`、`用户名`。
- 京东：兜底为 `用户名`。

同一研究项目、同一平台、同一标准化显示名会得到相同哈希，不受已登记源表头是 username 还是 nickname 类字段影响。跨研究项目、跨平台以及账号 ID 与显示名的哈希都不同。显示名关联属于弱伪名化，不是法律意义上的匿名化：昵称变化会把同一用户拆开，同名用户可能被合并。

原始账号 ID、用户名、昵称始终不会进入标准化或清洗输出、日志和摘要；已登记字段只在内存中用于确定性哈希。评论 ID、父评论 ID、URL、主页链接、IP、来源自带的 `哈希ID`、`用户身份` 和其它含义不明确的字段绝不作为身份来源。整个选择、标准化和哈希过程只使用确定性工具，不接入 AI。

首次处理研究项目时创建一次项目密钥，后续同项目复用。Windows 上密钥由 DPAPI 保护并保存在当前用户本地目录，不写入 Git、Excel、CSV、日志或摘要。该功能属于伪名化，不代表原始输入已经完全匿名；原始文件仍需限制访问。
